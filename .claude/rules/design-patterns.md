## Gang of Four Design Patterns

Apply patterns when the problem fits — not to demonstrate pattern knowledge.
Each pattern below is listed with where it applies in CFOAgent and what problem it solves.

---

### Strategy — LLM provider switching
**Problem:** the agent must work with Ollama in dev and Azure OpenAI/Anthropic in prod without `if/else` scattered through node code.
**Solution:** `BaseChatModel` is the strategy interface. `get_llm()` in `config.py` is the strategy selector. Nodes only call `llm.ainvoke()`.

```python
# Strategy interface: BaseChatModel (from LangChain)
# Concrete strategies: ChatOllama, AzureChatOpenAI, ChatAnthropic
# Context (node) calls the interface, never the concrete class

async def nl2sql_node(state: AgentState, llm: BaseChatModel, ...) -> dict:
    response = await llm.ainvoke(prompt_messages)  # strategy call
    return {"generated_sql": response.content}
```

### Factory Method — object creation without coupling
**Problem:** `graph.py` needs LLM and DB engine instances but shouldn't know how they're built.
**Solution:** `get_llm()` and `build_engine()` are factory functions. Callers receive a ready object.

```python
# agent/llm_factory.py
def get_llm() -> BaseChatModel:
    """Factory — returns the correct LLM for the current environment."""
    provider_map = {
        "ollama": _build_ollama,
        "azure_openai": _build_azure_openai,
        "anthropic": _build_anthropic,
    }
    builder = provider_map.get(settings.llm_provider)
    if not builder:
        raise ValueError(f"Unknown provider: {settings.llm_provider}")
    return builder()
```

### Chain of Responsibility — SQL pipeline
**Problem:** SQL generated by the LLM must pass through multiple validation steps (syntax check, blocked-statement check, row limit check) before execution.
**Solution:** each validator is a handler in a chain. Each handler either passes the request forward or short-circuits with an error.

```python
from abc import ABC, abstractmethod

class SqlHandler(ABC):
    def __init__(self) -> None:
        self._next: SqlHandler | None = None

    def set_next(self, handler: "SqlHandler") -> "SqlHandler":
        self._next = handler
        return handler

    @abstractmethod
    def handle(self, sql: str) -> tuple[bool, str]:
        ...

    def _pass_to_next(self, sql: str) -> tuple[bool, str]:
        if self._next:
            return self._next.handle(sql)
        return True, ""

class BlockedStatementHandler(SqlHandler):
    def handle(self, sql: str) -> tuple[bool, str]:
        if BLOCKED_PATTERN.search(sql):
            return False, "Blocked statement"
        return self._pass_to_next(sql)

class StackedQueryHandler(SqlHandler):
    def handle(self, sql: str) -> tuple[bool, str]:
        if ";" in sql.strip().rstrip(";"):
            return False, "Stacked queries not allowed"
        return self._pass_to_next(sql)

# Wired in graph.py — easy to add/reorder handlers
guardrail = BlockedStatementHandler()
guardrail.set_next(StackedQueryHandler())
```

### Facade — DB executor
**Problem:** executing a query involves engine setup, connection handling, cursor management, type coercion (Decimal → float), and result serialization. Nodes shouldn't know any of this.
**Solution:** `SqlExecutor` is a facade that hides SQLAlchemy complexity behind a single method.

```python
# db/executor.py
class SqlExecutor:
    """Facade over SQLAlchemy. Nodes call execute(), nothing else."""

    def __init__(self, engine: Engine) -> None:
        self._engine = engine

    async def execute(self, sql: str) -> list[dict]:
        async with self._engine.connect() as conn:
            result = await conn.execute(text(sql))
            return [self._serialize_row(row) for row in result.mappings()]

    def _serialize_row(self, row: RowMapping) -> dict:
        # Coerce Decimal → float, date → ISO string, etc.
        ...
```

### Template Method — base node with standard error handling
**Problem:** every LangGraph node needs the same boilerplate: validate preconditions, run logic, catch errors, write to state.
**Solution:** `BaseNode` defines the template. Subclasses implement `_run()`.

```python
# agent/nodes/base.py
from abc import ABC, abstractmethod

class BaseNode(ABC):
    async def __call__(self, state: AgentState) -> dict:
        try:
            if error := self._check_preconditions(state):
                return {"error": error}
            return await self._run(state)
        except Exception as exc:
            return {"error": str(exc)}

    def _check_preconditions(self, state: AgentState) -> str | None:
        """Return an error message if preconditions fail, else None."""
        return None

    @abstractmethod
    async def _run(self, state: AgentState) -> dict:
        ...

# agent/nodes/executor.py
class ExecutorNode(BaseNode):
    def __init__(self, executor: SqlExecutor) -> None:
        self._executor = executor

    def _check_preconditions(self, state: AgentState) -> str | None:
        if not state.get("sql_is_safe"):
            return "SQL did not pass guardrail"
        return None

    async def _run(self, state: AgentState) -> dict:
        rows = await self._executor.execute(state["generated_sql"])
        return {"query_result": rows}
```

### Singleton — Settings
**Problem:** `Settings` reads `.env` on construction — must not be instantiated multiple times.
**Solution:** module-level singleton. Import the instance, never the class.

```python
# config.py
settings = Settings()   # single instance, created at import time

# everywhere else
from config import settings   # import the instance
```

### Observer — audit logging
**Problem:** SQL generation and execution events need to be logged for audit without polluting node logic.
**Solution:** LangChain's `BaseCallbackHandler` is the observer. Register it at graph build time.

```python
from langchain_core.callbacks import BaseCallbackHandler

class AuditLogger(BaseCallbackHandler):
    async def on_llm_end(self, response, **kwargs) -> None:
        # log generated SQL to audit trail
        logger.info("generated_sql", sql=response.generations[0][0].text)

llm = get_llm().with_config(callbacks=[AuditLogger()])
```

---

### When NOT to use a pattern
- Don't apply a pattern unless the problem it solves is actually present in the code
- Don't add `AbstractBaseClass` to every class — use Python `Protocol` for structural typing instead
- Don't use `Singleton` for objects that are cheap to construct — use module-level instances only where construction has side effects (I/O, parsing)
